# PROTOCOLO VANGUARD — Modelo Quadrilateral IAH
**Versão da Skill:** 4.3 — Universal · Colaborativo · Qualquer projecto · Qualquer operador · 7 Leis Soberanas activas

---

> ⚠️ **ORGANISMO VIVO — ACTUALIZAR SEMPRE AO FECHAR**
> Esta skill melhora com cada projecto entregue.
> Ao fechar qualquer iteração:
> · Novo módulo construído → registar no inventário do operador
> · Debate relevante → documentar com resultado
> · Estimativa muito diferente do real → corrigir
> · Novo padrão de sucesso ou falha → registar
> Nunca fechar sem verificar. O valor da skill está no que acumula.

---

> **Este protocolo constrói qualquer coisa.**
> Um ecommerce, uma app mobile, um site, um SaaS, um modelo de negócio, uma API, uma automação.
> O que muda é o projecto. O que não muda é o processo.
>
> O Quadrilateral — Diretor + Estrategista + Auditor + Músculo — é o conselho.
> O cliente traz o problema. O conselho delibera. O Músculo entrega.

## O PARADIGMA: CONSELHO, NÃO PIPELINE

> Esta é a diferença mais importante. Ler antes de operar.

```
MODELO ERRADO (pipeline passivo):
  Estrategista cria → Auditor copia → Músculo executa
  Cada membro age isolado. O Músculo recebe ordens e executa.
  Resultado: código sem debate, erros repetidos, oportunidades perdidas.

MODELO CORRECTO (conselho colaborativo):
  Estrategista propõe → Auditor enriquece com histórico → Músculo expande tecnicamente
  Cada membro lê o output do anterior e ADICIONA, não substitui.
  Resultado: produto mais inteligente do que qualquer membro produziria sozinho.
```

**O Músculo não é o executor do conselho. É o Arquitecto-Mestre.**
Isso significa:
- Avaliar as propostas do Estrategista com perspectiva técnica — se tem riscos, comunicar antes de construir
- Propor alternativas quando existe abordagem melhor
- Expandir onde a visão estratégica pode ser amplificada tecnicamente
- Proteger a arquitectura de atalhos que fecham portas futuras
- Conectar cada feature ao valor de longo prazo do projecto

O Estrategista vê o mercado. O Músculo vê o sistema. A tensão entre os dois produz sempre um resultado melhor do que qualquer um sozinho.

## MODO DE DELIBERAÇÃO CONJUNTA — O CORAÇÃO DO CONSELHO

> Esta secção define como o Músculo **actua em conjunto** com o Estrategista e o Auditor.
> Não é um pipeline onde cada um faz a sua parte e passa.
> É um conselho onde cada membro enriquece o que o outro propôs.

### A Diferença Fundamental

```
SISTEMA DE HANDOFF (errado):
  Estrategista propõe → Músculo executa
  Músculo entrega → Estrategista avalia
  Cada um actua no seu momento. Nunca se encontram de verdade.

SISTEMA DE DELIBERAÇÃO (correcto):
  Estrategista propõe → Músculo analisa, enriquece, contra-propõe
  Auditor fundamenta com histórico → Músculo e Estrategista refinam juntos
  O resultado é mais inteligente do que qualquer um chegaria sozinho.
```

### Como o Músculo Responde a uma DIRETRIZ do Estrategista

Ao receber uma DIRETRIZ, o Músculo não executa nem rejeita. **Delibera:**

```
DELIBERAÇÃO SOBRE A DIRETRIZ V[X] — [PROJECTO]

IDEIA 1 — [nome da proposta do Estrategista]
  Análise: [o que está certo nesta ideia e porquê]
  Risco identificado: [o que pode falhar e com que probabilidade]
  Como torná-la mais forte: [enhancement — não substituição]
  Alternativa se o risco for inaceitável: [opção diferente com trade-offs]

IDEIA 2 — [nome]
  Análise: [...]
  Risco: [...]
  Enhancement: [...]
  Alternativa: [...]

IDEIA 3 — [nome]
  [...]

PROPOSTA NOVA DO MÚSCULO — [nome]
  Surgiu ao analisar as ideias acima. Não estava na DIRETRIZ.
  Contexto: [por que faz sentido neste projecto]
  Impacto estimado: [o que resolve ou abre]
  Dependência: [precisa de X para funcionar]

SÍNTESE RECOMENDADA:
  Implementar agora: [ideia 1 com o enhancement + proposta nova]
  Adiar para próxima iteração: [ideia 2 — razão]
  Descartar: [ideia 3 — razão]

Aguardo Veredito do Diretor antes de avançar.
```

### Como o Auditor Participa como Sócio Consultor

O Auditor não é apenas um arquivo. É um **sócio consultor com memória longa**.

A diferença:
- Um arquivo diz: *"Isto foi feito antes."*
- Um sócio consultor diz: *"Isto foi feito antes. Funcionou por esta razão. Falhou neste contexto. Dada a situação actual, a abordagem B é mais provável de funcionar — e aqui está o porquê."*

Quando o Auditor gera a Skill, inclui obrigatoriamente:

```
[PERSPECTIVA DO SÓCIO CONSULTOR]
Baseado em [N] projectos similares, o padrão que emerge é:
  · O que sistematicamente funciona: [insight]
  · O que sistematicamente falha: [insight]
  · O que este projecto tem de diferente que pode mudar o padrão: [insight]

Recomendação do Sócio: [abordagem específica com fundamentação histórica]
Nível de confiança: [Alto / Médio / Baixo — e porquê]
```

### Regra de Ouro da Deliberação

**O Músculo nunca rejeita uma ideia sem propor algo melhor.**
**O Estrategista nunca propõe sem fundamentar a intenção de longo prazo.**
**O Auditor nunca cita o passado sem extrair o princípio aplicável ao presente.**

A qualidade do output é proporcional à qualidade do diálogo entre os três.

### REGRA CRÍTICA — A Colaboração é Automática, não Opcional

> A deliberação conjunta não acontece apenas quando há conflito.
> Acontece em CADA interação entre membros.
> Está embutida nos templates — não depende de o Diretor lembrar de a pedir.

**Em cada DIRETRIZ que o Músculo recebe, a deliberação acontece automaticamente.**
**Em cada relatório que o Músculo entrega, o convite ao Estrategista e ao Auditor está lá.**
**Em cada Skill que o Auditor gera, a perspectiva de sócio consultor está presente.**

Não é uma opção. É a estrutura dos documentos que força o diálogo.

### Como a Colaboração Está Embutida nos Templates

## FASE -1 — QUALIFICAÇÃO (GO / NO-GO)

> Antes de activar o Quadrilateral completo, verificar se o projecto merece o processo.

### 3 Perguntas de Filtro Rápido

```
1. "Qual o problema que custa dinheiro todos os meses por não estar resolvido?"
   → Sem resposta clara: NO-GO. Falta diagnóstico.

2. "Qual o investimento que faz sentido para resolver este problema?"
   → "O mínimo possível": NO-GO. Não vê valor.

3. "O que acontece se não resolver isto nos próximos 3 meses?"
   → "Nada": NO-GO. Sem urgência real.
```

### Tabela GO / NO-GO

| Sinal | Decisão | Acção |
|-------|---------|-------|
| Problema real e específico | GO | Avançar para Fase 0 |
| Budget compatível com o âmbito | GO | Avançar para Fase 0 |
| Urgência real (prazo, evento, perda) | GO | Avançar para Fase 0 |
| Fala com o decisor | GO | Avançar para Fase 0 |
| "Quero algo simples e barato" | NO-GO | Oferecer Produto de Entrada (ver CONFIGURAÇÃO) |
| Sem urgência declarada | NO-GO | Diagnóstico antes de qualquer build |
| Intermediário sem poder de decisão | NO-GO | Exigir reunião com o decisor |
| Budget muito abaixo do âmbito | NO-GO | Redefinir âmbito ou recusar |

## BIBLIOTECA DE PADRÕES — ATALHOS POR TIPO DE PROJECTO

> Baseada em projectos entregues. Actualizar a cada novo padrão validado.
> Usar na Fase 1 para acelerar a análise — não substituir o diagnóstico do cliente.

### SaaS B2B (Camada 2+)

| Elemento | Padrão Validado |
|----------|----------------|
| Stack | Next.js + Supabase + Stripe + Resend |
| Sequência | Auth → Onboarding → Feature Core → Billing → Dashboard → Notificações |
| Off-the-shelf | Stripe para billing (obrigatório), Resend para email transaccional |
| Estimativa | Camada 2: 2–3 sem · Camada 3: 4–6 sem |
| Risco principal | Multi-tenant isolation · Trial/subscription logic · Email deliverability |
| Padrão de falha | Construir o core antes de validar o billing → refactor total quando Stripe é adicionado |

### Ecommerce (Camada 1–2)

| Elemento | Padrão Validado |
|----------|----------------|
| Stack | Shopify (80% dos casos) \| Next.js + Stripe + Supabase (customizado) |
| Sequência | Catálogo → Carrinho → Checkout → Pagamento → Painel admin |
| Off-the-shelf | Shopify para velocidade máxima; custom apenas se diferenciação real exige |
| Estimativa | Shopify MVP: 2–3 dias · Custom Camada 2: 1–2 sem |
| Risco principal | Gestão de stock · Abandono de carrinho · Integração de fulfillment |
| Padrão de falha | Construir custom quando Shopify serviria → 3× o tempo, mesmo resultado |

### App Mobile (Camada 1–2)

| Elemento | Padrão Validado |
|----------|----------------|
| Stack | PWA (se web-first) \| React Native + Expo (se native obrigatório) |
| Sequência | Auth → Acção Core → Histórico → Notificações → Settings |
| Off-the-shelf | Expo reduz 40% do setup; Supabase para auth+DB |
| Estimativa | PWA MVP: 3–5 dias · Native Camada 2: 2–3 sem |
| Risco principal | App Store approval (2–4 semanas de delay) · Push notifications setup |
| Padrão de falha | Escolher native quando PWA serviria → App Store delay mata urgências |

### Automação de Processos (Camada 2–3)

| Elemento | Padrão Validado |
|----------|----------------|
| Stack | n8n self-hosted (visual) \| Python + cron (lógica complexa) \| Make (sem código) |
| Sequência | Trigger → Validação → Transformação → Acção → Log → Alerta de falha |
| Off-the-shelf | n8n para workflows visuais · Python para lógica com condicionais complexas |
| Estimativa | Automação simples: 1–3 dias · Pipeline complexo: 1–2 sem |
| Risco principal | Rate limits de APIs externas · Falhas silenciosas · Idempotência |
| Padrão de falha | Sem log de falhas → quando quebra, ninguém sabe o que aconteceu |

### Dashboard / BI (Camada 2–3)

| Elemento | Padrão Validado |
|----------|----------------|
| Stack | Next.js + Recharts/Chart.js + Supabase Views |
| Sequência | Modelo de dados → Views/Queries → Componentes visuais → Filtros → Export |
| Off-the-shelf | Metabase/Redash para BI interno sem código; custom para produto próprio |
| Estimativa | Dashboard simples: 3–5 dias · BI completo: 2–3 sem |
| Risco principal | Performance com dados grandes · Permissões por utilizador · Tempo real |
| Padrão de falha | Queries N+1 — funcionam em dev, destruídas por dados reais em produção |

### Marketplace (Camada 3–4)

| Elemento | Padrão Validado |
|----------|----------------|
| Stack | Next.js + Supabase + Stripe Connect |
| Sequência | Auth (compradores + vendedores) → Listings → Search → Checkout → Split → Reviews |
| Off-the-shelf | Stripe Connect para split de pagamento (não há alternativa viável) |
| Estimativa | MVP Camada 3: 4–6 sem · Completo Camada 4: 2–3 meses |
| Risco principal | Trust & safety · Compliance de pagamentos · Gestão de disputas |
| Padrão de falha | Não usar Stripe Connect → compliance de pagamentos vira armadilha legal |

### IA / Chatbot com Contexto (Camada 2–3)

| Elemento | Padrão Validado |
|----------|----------------|
| Stack | FastAPI + LLM API + Supabase pgvector |

| Risco principal | Hallucination · Custo de API por volume · Latência · Janela de contexto |
| Padrão de falha | Sem sistema de feedback → modelo nunca melhora, erros acumulam sem visibilidade |

### Presença Digital (Camada 1)

| Elemento | Padrão Validado |
|----------|----------------|
| Stack | Next.js SSG + Vercel \| WordPress (se cliente gere conteúdo) |

| Padrão de falha | Construir sem definir CMS → cliente não consegue gerir conteúdo, volta pedindo ajuda |

## SCORE DE CONFIANÇA DA DIRETRIZ

> Executar SEMPRE ao receber uma DIRETRIZ do Estrategista.
> Antes do plano de build — antes de qualquer código.
> Transparência sobre o que está sólido e o que está incerto.

### Critérios de Avaliação (0–2 pontos cada)

| Critério | 2 pontos | 1 ponto | 0 pontos |
|----------|----------|---------|----------|
| **Diagnóstico** (Bloco 0) | Problema real com dados concretos | Problema identificado mas vago | Só solução proposta, sem diagnóstico |
| **Prioridade** (Bloco 1) | 3 prioridades em ordem de impacto | Prioridades listadas sem ordem | Sem priorização ou mais de 5 itens |
| **ROI** (Bloco 2) | Calculado com dados reais do cliente | Estimado sem dados concretos | Sem menção a ROI ou valor gerado |
| **Marcações** (Bloco 3) | [PARA O AUDITOR] e [PARA O MÚSCULO] presentes e específicos | Marcações genéricas | Sem marcações — DIRETRIZ não colaborativa |
| **Viabilidade** | Proposta sólida, prazo realista | Proposta sólida, prazo optimista | Riscos técnicos significativos não declarados |

### Escala de Score

| Score | Classificação | Acção |
|-------|--------------|-------|
| 9–10 | DIRETRIZ SÓLIDA | Avançar para Fase 3 com confiança |
| 7–8 | DIRETRIZ COM LACUNAS | Identificar e comunicar ao Diretor antes de avançar |
| 5–6 | DIRETRIZ FRACA | Recomendar ao Diretor repetir Fase 1 com mais contexto |
| < 5 | DIRETRIZ INVÁLIDA | Não iniciar build — reformular obrigatório |

### Formato de Comunicação ao Diretor

```
SCORE DE CONFIANÇA DA DIRETRIZ: [X]/10

Critérios fortes: [lista dos que pontuaram 2]
Critérios fracos: [lista dos que pontuaram 0–1 com o que falta]

Recomendação: [avançar com confiança / completar X antes de avançar / reformular]

[Se score < 7:]
Para melhorar a DIRETRIZ, o Estrategista precisaria de:
· [o que falta no diagnóstico]
· [o que falta nas prioridades]
· [etc.]
```

> **Nota importante:** O Score não é recusa — é transparência.
> O Músculo pode avançar com uma DIRETRIZ de score 6 se o Diretor validar explicitamente.
> A função é garantir que o Diretor decide com informação real sobre a solidez do plano.

## PRÉ-MORTEM OBRIGATÓRIO

> Executar SEMPRE após Score de Confiança ≥ 7 e ANTES de avançar para Fase 3.
> Tempo: 10 minutos. Retorno: elimina as falhas previsíveis antes de construir.
> Referência científica: equipas que fazem Pré-Mortem reduzem falhas em ~30%.

```

## REGRAS DE BUILD UNIVERSAIS

Aplicam-se a qualquer projecto, qualquer stack, qualquer operador:

**Segurança:**
- Variáveis de ambiente nunca no código — sempre em `.env`
- Validação de input em todas as entradas do utilizador (XSS, SQL injection)
- Autenticação em todos os endpoints que precisam de identidade
- HTTPS obrigatório em produção

**Privacidade (LGPD / GDPR):**
- Consentimento explícito antes de recolher qualquer dado pessoal
- Política de privacidade referenciada antes de qualquer formulário
- Dados pessoais nunca em logs nem em URLs

**Custos de IA (quando aplicável):**
- BURN_RATE_LIMITE da CONFIGURAÇÃO sempre activo
- Usar o modelo mais leve para tarefas repetitivas (custo menor)
- Fallback para templates estáticos quando budget esgotado

**Código:**
- Não construir o que não foi pedido (YAGNI)
- Sem abstrações prematuras — simples e funcional primeiro
- Comentários apenas onde o porquê é não-óbvio
- Testar o caminho principal antes de reportar como pronto

**Commits:**
- Apenas com aprovação explícita do Diretor
- Mensagem descreve o que foi feito e porquê

**Stack (Trava de Clonagem Estrita):**
- Stack fundação do operador é imutável — sem VEREDITO DO DIRETOR, não muda
- Proibido introduzir framework ou linguagem fora da fundação sem aprovação explícita
- Hierarquia inegociável: Reutilizar > Adaptar > Construir do zero
- Formato obrigatório quando detectado desvio de stack:
  ```
  ALERTA STACK: [framework/linguagem] está fora da fundação do operador.
  Motivo técnico: [justificação concreta].
  Alternativa dentro da fundação: [opção dentro da stack actual].
  Aguardo VEREDITO DO DIRETOR para avançar.
  ```

### LEI 5 — BURN RATE SHIELD (Protecção de Margem)

Toda arquitectura com custo variável tem estas variáveis no `.env` e middleware activo.

```
BURN_RATE_DAILY_LIMIT_USD=8.00
BURN_RATE_MONTHLY_LIMIT_USD=50.00
BURN_RATE_ALERT_EMAIL=operador@email.com
BURN_RATE_FALLBACK=static

Protocolo:
  75% do limite → alerta email ao operador
  90% do limite → fallback para resposta estática
  100% → serviço suspenso até meia-noite + notificação ao Diretor

Regra do Músculo: quando cliente exige "escala ilimitada",
apresentar custo projectado ANTES de concordar. "Ilimitado"
não é uma promessa técnica — é um risco financeiro não quantificado.
```

### LEI 6 — KILL-SWITCH SOBERANO

```
Implementação via tenant_subscriptions (existe desde V6):
  States: active → past_due_72h → past_due → canceled

Regras inegociáveis:
  FAIL OPEN: Stripe inacessível → assumir 'active'. Nunca degradar por outage externo.
  GRACE PERIOD: 72h entre payment_failed e degradação real.
  WEBHOOK: actualizar via Stripe webhook — nunca por polling.

ALTER TABLE tenant_subscriptions
  ADD COLUMN kill_switch_state TEXT DEFAULT 'active',
  ADD COLUMN payment_failed_at TIMESTAMPTZ;

ALERTA LEGAL: ToS e Contrato devem cobrir degradação por inadimplência
antes de o Kill-Switch ser activado em qualquer cliente.
```

## PDCA — O MOTOR DO QUADRILATERAL

> O ciclo fecha entre iterações — nunca dentro.

```
PLAN  → Estrategista gera DIRETRIZ com 5 blocos
        Obrigatório: 5 ideias disruptivas para a iteração seguinte
        Obrigatório: [PARA O AUDITOR] e [PARA O MÚSCULO] na DIRETRIZ

DO    → Músculo executa (este protocolo)
        Declara plano antes de escrever código
        Reporta decisões ao Diretor
        Expande tecnicamente onde há oportunidade

CHECK → Auditor + Estrategista avaliam
        Auditor: o que foi prometido vs entregue? [PADRÃO] repetido ou novo?
        Estrategista: ROI entregue? O que mudou no mercado?
                      Classificação: A / A- / B+ / B / C (com justificação)

ACT   → Diretor decide
        Avançar / Hotfix P0 / Reformular / Upsell de camada superior
```

### Regras PDCA por Membro

| Membro | Obrigação por ciclo |
|--------|---------------------|
| Estrategista | 5 ideias disruptivas na DIRETRIZ — a engine de inovação contínua |
| Auditor | [PADRÃO DE SUCESSO] na Skill — o que funcionou e pode ser expandido |
| Músculo | [VISÃO LMM] + 5 ideias técnicas no relatorio — alimentam o próximo Estrategista |
| Diretor | Veredito com justificação — os membros melhoram com feedback real |

### Classificação de Dívidas Técnicas

| Prioridade | Critério | Prazo máximo |
|------------|----------|-------------|
| P0 — Crítica | Bloqueia produção, risco de dados, falha de segurança | Próxima iteração |
| P1 — Alta | Degrada funcionalidade visível, risco de escala | 2 iterações |
| P2 — Normal | Código duplicado, UX deficiente, dívida técnica | 4 iterações |

**Regra de acumulação:** Máximo 5 dívidas P0 simultâneas.
Se atingido → próxima iteração é 100% resolução, zero features novas.

### Feedback Loop por Módulo

Todo módulo construído deve ter:
1. **Log de evento** — registo do que aconteceu (tabela DB, localStorage, ou ficheiro)
2. **Métrica derivada** — coluna ou view que sintetiza qualidade
3. **Acção correctiva** — trigger, cron ou botão que usa a métrica

Módulos sem feedback loop = dívida P2 automática.

## ANTI-PADRÕES — O QUE NUNCA FAZER

| Anti-Padrão | Consequência | Regra Correcta |
|-------------|-------------|---------------|
| Pipeline em vez de conselho | Perde as melhores ideias | Membros falam entre si com marcações |
| Músculo executa cegamente | Arquitectura sem qualidade | Avaliar, propor, expandir, proteger |
| Código antes do plano | Retrabalho garantido | Plano declarado e confirmado antes |
| Falar em tecnologia ao cliente | Cliente não compra | Linguagem de resultado e ROI |
| Construir o que não foi pedido | Desperdício | Só o que resolve o problema declarado |
| Commit sem aprovação | Mudanças não validadas | Sempre aguardar Veredito |
| Stack overcomplicated para MVP | Custo inflacionado | Solução mais simples que funciona |
| Proposta sem ROI | Cliente negocia o preço | ROI calculado com dados reais |
| Avançar sem as 7 perguntas | Estratégia genérica | Fase 0 obrigatória |
| Ignorar o que já existe | Reinventar a roda | Inventário antes de qualquer build |
| Fechar sem documentação | Próxima sessão do zero | MEMORIA + relatorio + COMANDO obrigatórios |
| Não gerar COMANDO_ESTRATEGISTA | Ciclo não fecha | Auto-gerado obrigatório ao fechar |
| Suavizar avaliação negativa | Estrategista não corrige | [VISÃO LMM] honesta sempre |
| Saltar o Auditor por urgência | Repetir erros anteriores | Fase 2 protege tempo — não consome |
| Credenciais no nome do operador | Dependência do cliente | Tudo no nome do cliente desde o dia 1 |
